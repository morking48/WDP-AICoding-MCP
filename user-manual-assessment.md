## 二、诚实评估：是否需要添加提示？

### 评估原则
- **用户手册应该简洁**，只告诉用户"如何输入"
- **不应该暴露内部实现细节**（如业务场景、API模式）
- **AI 负责理解复杂逻辑**，用户只需描述需求

### 具体评估

#### 1. 业务场景（如视频周界、路径漫游）

**建议**：❌ **不需要添加**

**理由**：
- 这些是 AI 内部使用的场景分类
- 用户只需用自然语言描述需求（如"我要做视频监控"）
- AI 会根据 wdp-intent-orchestrator 自动匹配场景
- 添加反而增加用户认知负担

#### 2. API 模式（如 POI弹窗交互、路径漫游）

**建议**：❌ **不需要添加**

**理由**：
- 这些是 AI 内部使用的 API 调用模式
- 用户不需要知道具体的 API 调用顺序
- AI 会根据意图编排自动生成调用链路

#### 3. 整链路需求（如车辆巡检需要多个 skill）

**建议**：⚠️ **可以添加一句提示**

**理由**：
- 用户可能不知道复杂需求需要多轮处理
- 可以简单提示"复杂需求可能需要多轮对话完成"
- 但不需要详细说明涉及哪些 skill

**建议添加位置**：场景3（单个需求描述解析）或场景4（开始编码）的说明部分

**建议添加内容**：
```markdown
**复杂需求提示**：
如果需求涉及多个功能（如车辆巡检、跟拍、多系统联动），AI 会自动进行意图编排和任务拆解，可能需要多轮对话完成。请按 AI 提示逐步提供信息。
```

#### 4. 长流程任务判断（超过5步、跨 skill）

**建议**：❌ **不需要添加**

**理由**：
- 这是 AI 自动判断的，用户不需要关心
- 用户手册第 181-194 行已有"续接进度"场景
- 用户只需知道"长任务可以续接"即可

#### 5. 清理链路意识（创建动作需要对应清理）

**建议**：⚠️ **可以添加一句提示**

**理由**：
- 用户可能不知道需要清理资源
- 可以提示"涉及创建、注册的功能，退出时需要清理"
- 帮助用户理解为什么 AI 会询问退出逻辑

**建议添加位置**：对象类别说明之后

**建议添加内容**：
```markdown
### 清理链路提示

如果需求涉及创建对象、注册事件、启动动画等操作，**退出页面或卸载时需要对应的清理动作**（如删除对象、解绑事件、停止动画）。请在需求描述中说明是否需要自动清理，或提供手动清理方式。
```

## 三、最终建议

### 推荐修改（可选）

**修改 1**：在场景3或场景4添加复杂需求提示（1-2句话）

**修改 2**：在对象类别说明后添加清理链路提示（1段）

### 修改后的效果

- 用户知道复杂需求可能需要多轮对话
- 用户知道需要考虑资源清理
- 不增加过多认知负担
- 保持手册简洁

### 不推荐的修改

- ❌ 添加详细的业务场景列表
- ❌ 添加 API 模式说明
- ❌ 添加长流程判断标准
- ❌ 添加意图编排流程说明

---

## 四、结论

**当前用户手册 ✅ 基本满足需求**

**可选优化**：
1. 添加一句复杂需求提示
2. 添加一句清理链路提示

**优先级**：低（锦上添花，非必需）

**建议**：如果用户没有反馈理解困难，可以不修改。当前手册已经涵盖了用户需要知道的核心信息。
