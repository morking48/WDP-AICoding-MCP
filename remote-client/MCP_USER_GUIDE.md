# WDP MCP 用户操作手册

> 复制以下指令，替换成你的需求后，直接粘贴给支持 MCP 的 AI 客户端即可使用。

---

## 适用对象

本手册适用于通过 MCP 访问 WDP 知识服务的用户。

这份手册的目标是：

- 提高需求命中率
- 减少错误方法、错误参数、错误调用顺序
- 让系统先确认能力和参数，再进入编码

---

## MCP 模式下不要省去的关键锚点

MCP 比本地直读模式多了一层 tool 调度，但下面这些关键 skill 和注释不要省去。为了尽量保持和本地手册一致，建议你在提问时继续显式提醒它们：

- `wdp-entry-agent`
  负责入口路由、执行流程确认和能力边界判断。
- `wdp-intent-orchestrator`
  负责需求梳理、缺参检查和《系统意图与架构设计报告》输出。
- `wdp-context-memory`
  适合长流程任务的阶段状态管理，不要在多轮开发里默默省去。
- `official-*.md`
  是具体 API 方法名、参数名、代码示例的真值来源，不能只靠 skill 描述直接写代码。

MCP 模式下的等价理解：

- 本地模式是你手动要求读取某个 skill 文件。
- MCP 模式是你显式提醒系统先按这些 skill 的职责执行，再通过 tool 去检索和读取知识库。
- 如果当前没有命中正确知识，不要让系统跳过这些锚点直接自由编排。

## 第一步：启动新任务

每次开始新的 WDP 任务，先发送：

请直接复制下面整段内容：

```可复制指令
我要开始一个新的 WDP 开发任务。

请先按 wdp-entry-agent 的规则完成需求路由、参数检查和能力匹配，再决定后续步骤。
如果当前信息不足，请先向我追问，不要直接编写代码。
如果涉及具体 WDP API，生成代码前必须先确认对应的 official-*.md 文档依据。
```

## 第二步：需求梳理（必须）

如果你还在梳理需求，或者需求文档比较长，先发送：

请直接复制下面整段内容，只替换 `【需求描述】`：

```可复制指令
请先按 wdp-intent-orchestrator 的方式结构化梳理下面这个 WDP 需求，并输出《系统意图与架构设计报告》。

需求描述：
【替换为你的需求描述或需求文档内容】

要求：
1. 先梳理，不要直接写代码
2. 如果缺少关键参数，请直接追问我
3. 如果涉及坐标、位置，请指出需要哪些具体参数
4. 如果涉及对象操作，请根据对象类别明确指出需要 Eid、entityName、customId、seedId、nodeId 或 featureId 中的哪一种标识，并优先检索适配此类 Id 的相关 skill
5. 只有确认到可用的 WDP 能力后，再进入编码阶段
```

如果你已经有需求文档路径，也可以这样发：

请直接复制下面整段内容，只替换 `【需求文件路径】`：

```可复制指令
请先按 wdp-intent-orchestrator 的方式结构化梳理这个 WDP 需求文件，并输出《系统意图与架构设计报告》。

需求文件路径：
【替换为你的需求文件路径】

要求：
1. 先梳理，不要直接写代码
2. 如果缺少关键参数，请直接追问我
3. 先明确主技能域，例如 BIM、GIS、相机、覆盖物、事件、图层、材质或空间理解
4. 如果匹配到对象操作，请明确需要 Eid、entityName、customId、seedId、nodeId 或 featureId 中的哪一种标识
5. 只有确认到可用的 WDP 能力后，再进入编码阶段
```

## 第三步：开始编码

确认需求报告后，再发送：

请直接复制下面整段内容：

```可复制指令
请基于已经确认的需求和报告，开始执行编码工作。

要求：
1. 先按 wdp-entry-agent 确认执行流程，再开始写代码
2. 这是一个长流程任务，请使用 wdp-context-memory 管理阶段状态，并保持阶段性确认
3. 代码生成前必须先确认对应的 official-*.md 文档依据
4. 需要人工确认的部分，随时中断并咨询我
5. 如果当前信息不足，请先追问，不要猜方法名或参数名
```

如果你已经有《系统意图与架构设计报告》，可以这样发：

请直接复制下面整段内容，只替换 `【报告路径】` 和 `【补充说明】`：

```可复制指令
请读取并执行这份《系统意图与架构设计报告》对应的开发任务。

报告路径：
【替换为你的报告路径】

补充说明：
【替换为你的项目路径、现有代码情况或其他上下文】

要求：
1. 先按 wdp-entry-agent 确认执行流程，再开始写代码
2. 这是一个长流程任务，请使用 wdp-context-memory 管理阶段状态
3. 代码生成前必须先确认对应的 official-*.md 文档依据
4. 缺少关键参数时先追问我
5. 需要人工确认的部分随时中断咨询
```

## 推荐补充信息

为了减少来回追问，建议你尽量一次性补充以下信息：

- 场景类型：BIM / GIS / 相机 / 覆盖物 / 事件 / 图层 / 材质 / 空间理解
- 当前阶段：从零开始，还是在现有工程上修改
- 基础环境：WDP 服务 URL、Order、容器 ID
- 目标对象：Eid、entityName、customId、seedId、nodeId、featureId、图层名、模型名
- 业务数据：经纬度、GeoJSON、点位列表、弹窗内容、路径数据
- 目标效果：高亮、漫游、定位、点击交互、信息窗、图层控制、热力图

如果涉及对象操作，建议至少提供以下任意一种有效标识：

- `Eid`：实体 Id，例如 `827604541699522560`
- `entityName`：实体名称或场景大纲中的模型名称，例如 `机器人001`、`XXXX.fbx`
- `customId`：为模型自定义的 Id
- `seedId`：平台官方模型或 DaaS 上传模型编号
- `nodeId`：构件 Id 或底板单体 Id
- `featureId`：GIS 矢量图层中的要素编号，常用于要素高亮或定位

建议原则：

- BIM / 构件 / 房间 / 单体控制类需求，优先补 `Eid`、`nodeId`、`entityName`
- 模型资源或 DaaS 相关需求，优先补 `customId`、`seedId`
- GIS 矢量图层相关需求，优先补 `featureId`

如果你怀疑系统没有抓到对的知识，不要只说“继续写”。更有效的做法是补一句：

```可复制指令
如果当前没有直接命中，请不要跳过。

请继续执行以下动作：
1. 重新按 wdp-entry-agent 判断技能路由
2. 列出最接近的候选 skill
3. 列出对应的 official-*.md 文档
4. 如果是对象操作，请明确我还缺 Eid、entityName、customId、seedId、nodeId 或 featureId 中的哪一种标识
5. 在没有确认依据前，不要直接生成 WDP API 代码
```

```可复制指令
需求描述：
【替换为你的需求】

已知信息：
- 场景类型：【BIM / GIS / 其他】
- 当前阶段：从零开始
- URL：【替换】
- Order：【替换】
- 容器 ID：【替换】

要求：
1. 先按 wdp-entry-agent 梳理需求和技能路由，不要直接写代码
2. 如果还缺信息，请直接追问我
3. 再按 wdp-intent-orchestrator 输出结构化梳理结果

```

请直接复制下面整段内容：

```可复制指令

1. 不要沿用刚才未经确认的假设

4. 如果没有确认到 official-*.md 文档依据，不要直接编写 WDP API 代码
```

### 模板 5：限制系统不要猜 API

请直接复制下面整段内容：

```可复制指令

2. 不要猜参数名、枚举值或返回结构

```

```可复制指令

4. 如果需要，请扩大搜索范围到相关模板、资源文件和参考文件
5. 如果仍然无法确认，请明确告诉我还缺哪些参数或对象标识
6. 在没有确认依据前，不要编写具体 WDP API 代码
```

| 需求类型 | 目标 Skill | 推荐描述方式 |
|---------|-----------|-------------|
| 启动/接入失败 | `wdp-api-initialization-unified` | “WDP 页面初始化失败，请先按 wdp-entry-agent 检查初始化链路和参数缺失情况” |
| 相机/漫游问题 | `wdp-api-camera-unified` | “我要做相机聚焦/飞行/漫游，请先确认相机能力、前置条件和 official-*.md 依据” |
| 实体检索/显隐/删除 | `wdp-api-entity-general-behavior` | “我要控制实体显隐/删除/检索，请先确认需要提供 Eid、entityName、customId、seedId 还是 nodeId” |
| 覆盖物（POI/Window/热力图） | `wdp-api-coverings-unified` | “我要做 POI/Window/热力图，请先确认需要哪些坐标、对象或数据结构” |

## 重要提醒

1. 必须先梳理需求再编码，避免系统在低置信度下自行编排错误方法。
2. 如果系统先追问你，不代表失败，而是在避免错误 API 和错误参数。
3. 代码生成前必须确认 official-*.md 文档依据，不能只靠 skill 描述直接写代码。
4. 如果你看到系统开始猜方法名、猜参数名，要立刻打断并要求重新确认依据。
5. 如果任务较长，建议明确要求使用 `wdp-context-memory` 做阶段状态管理。
6. 如果当前回答没有提到 `wdp-entry-agent`、`wdp-intent-orchestrator` 或 `official-*.md`，要警惕它可能跳过了关键步骤。

## 常见误区

### 误区 1：只说“帮我做个页面”

不推荐：

```不推荐示例
帮我做个 3D 页面。
```

推荐：

```推荐示例
我要做一个 WDP BIM 页面，目标是展示大楼并支持点击楼层高亮房间。当前是从零开始，请先按 wdp-entry-agent 和 wdp-intent-orchestrator 梳理需求，并告诉我还缺哪些参数。
```

### 误区 2：系统追问时强行要求直接写代码

不推荐：

```不推荐示例
别问了，直接写。
```

推荐：

```推荐示例
请先告诉我当前还缺哪些关键参数，我补充后你再继续写代码。
```

### 误区 3：把业务名词直接当成 API 名

不推荐：

```不推荐示例
帮我调用某某高亮接口。
```

推荐：

```推荐示例
我要实现构件高亮，请先确认这个能力在当前 WDP 知识库中对应哪一类 skill 和哪份 official-*.md 文档。
```

## 一个最短可用流程

1. 先发送“启动新任务”模板，提醒它按 `wdp-entry-agent` 开始。
2. 再发送“需求梳理”模板，提醒它按 `wdp-intent-orchestrator` 输出报告。
3. 按系统追问补齐 URL、Order、对象信息、坐标或数据。
4. 等系统确认目标 skill 和 `official-*.md` 文档依据后，再发送“开始编码”模板。
5. 如果怀疑知识没有抓全，使用“怀疑知识库漏抓或命中不准”模板。
6. 如果运行报错，再用“修复运行问题”模板继续。
